# ADR: инвариант #7 — от «exit-код = решение» к decision stream (резидентная консоль)

> Статус: ПРИНЯТО (Гейт-1 Этапа 2 Фазы 2, 2026-07-12; ратифицировано Ревьюером и Капитаном).
> Контекст: Console v2 Фаза 2 — консоль перестаёт выходить по решению
> (`meta/docs/CONSOLE-V2-SPEC.md` §5, спека `2026-07-12-console-v2-phase2-s2-residency.md`).

## Проблема

Инвариант AGENTS.md #7 («UI → stderr, machine decision → stdout; exit codes:
approved / rejected / expired / aborted / no-tty») — контракт one-shot эпохи:
одно решение = один процесс = один exit-код. Резидентная консоль принимает много
решений за сессию — exit-код физически не может нести «решение». Требования
«резидентность» (ТЗ §5, ратифицировано) и «exit-коды не ослаблять» (ТЗ §4.4)
в буквальном чтении несовместимы; побеждает резидентность, потому что именно
её Капитан заказал как продуктовое направление.

## Решение

1. **Machine decision → stdout остаётся, но per-decision и только не-в-TTY.**
   Каждое терминальное решение (`executed`/`denied`/`expired`/`failed`) печатает
   на stdout ту же JSON-строку, что и сегодняшний `decision_line`
   (`{"decision":"approved","tx_hash":"0x…"}`), в момент решения —
   newline-delimited stream. **Условие: stdout не TTY** (`std::io::IsTerminal`,
   тот же трейт, каким инвариант #4 проверяет stdin). Причина условия: в
   `docker exec -it` stdout — тот же терминал, что alternate-screen на stderr;
   запись посреди сессии легла бы поверх кадра TUI. Сегодняшний
   `restore_then_announce` (main.rs) борется ровно с этим — говорить в stdout
   можно только когда это не экран. Интерактивный человек видит outcome-баннер
   в UI; машина в пайпе получает поток строк.
2. **Exit-коды становятся сессионными:** `aborted` (quit) / `no-tty` / `fatal` /
   `upgrade` — все четыре сохраняются с прежними значениями. Per-decision коды
   (`approved`/`rejected`/`expired`) уходят — их смысл переехал в stream.
3. AGENTS.md #7 переписывается: «machine decisions → stdout (one JSON line per
   decision, emitted only when stdout is not a TTY); exit codes report the
   session end: aborted / no-tty / fatal / upgrade».

## Почему так, а не иначе

- **Дух инварианта сохранён полностью**: машине есть что парсить (стало даже
  богаче — поток вместо одного кода), UI не мешает машинному каналу.
- **Ничего не строим впрок**: tty-проверка — одна строка существующим трейтом;
  формат строки не меняется.
- **Отвергнуто: one-shot режим за флагом** (обратная совместимость для скриптов).
  Второй кодовый путь и тестовая матрица под сценарий, которого нет у реальных
  потребителей: единственное место в организации, читающее exit-код консоли, —
  `mcp/tests/e2e/console.py:107` (приёмка), и она живёт против пина
  `CONSOLE_IMAGE=v0.1.0` в `mcp/Dockerfile.wallet:19` — обновится на релизном
  поезде (ТЗ §8) вместе с бампом образа, не раньше. Прод-код агента
  (`mcp/src/rustok_mcp/handlers.py`) от exit-кода консоли не зависит вовсе.
- **Что изменило бы решение**: внешний потребитель exit-кодов вне нашего
  контроля. Такого нет — консоль ставится только нашим образом.

## Последствия

- `Phase::Resolved` / `ExitOutcome` уходят из модели; классификация терминальных
  исходов остаётся для баннера и decision line.
- mcp e2e-приёмка переписывается при бампе `CONSOLE_IMAGE` на v0.2.0 (релизный
  поезд §8) — отмечено в спеке Этапа 2 как downstream, не в скоупе PR.
- Инвариант #4 (no-TTY → view-only + отдельный exit-код) не меняется.
