# Отчёт: Console v2 Фаза 2, Этап 2 — резидентность + nav-частично + From→To

Спека: `.claude/specs/2026-07-12-console-v2-phase2-s2-residency.md`. ADR:
`.claude/decisions/2026-07-12-invariant-7-decision-stream.md`. Ветка
`feat/console-v2-phase2-s2-residency` (локально, НЕ запушена).

## Сделано — по пунктам спеки

1. **Резидентность**: терминальный ответ закрывает карточку, ставит outcome-баннер,
   возвращает в очередь; `Phase::Resolved`/`ExitOutcome` удалены, `decision_kind()` —
   классификатор (та же таблица маппинга, тест сохранён 1:1). Fatal — как был.
2. **Decision stream** (ратифицированный §7): JSON-строка на stdout per-decision,
   **только когда stdout не TTY** (`IsTerminal`); формат идентичен старому
   `decision_line`. Exit-коды сессии сохранили v0.1-значения: fatal=1, upgrade=2,
   no-tty=3, aborted=6 (тест пиннит значения). AGENTS.md #7 переписан + ADR.
3. **PIN-lockout**: `Locked`/`BadPin(0)` на approve-пути закрывают карточку и ставят
   амберный баннер «PIN locked — pending items were denied. Retry in ~Ns.» (§4-честный
   текст — Executing не хоронится).
4. **Client proto 2 + context**: `PROTO_VERSION=2`, `Request::Context` шлётся один раз
   сразу после auth-ok (до первого поллинга — контеншн-политика тривиально выполнена),
   `WalletContext` хранится в Model сессионно, балансы парсятся полным типом (Этап 5
   протокол не тронет).
5. **Двух-блочный From→To**: вертикальный стек ВНУТРИ `priority_lines()` (гейт
   `priority_fields_fit` считает каждую строку — side-by-side ушёл бы из-под гейта),
   адреса полные verbatim EIP-55. Деградация (wallet_locked) → To-only карточка
   Фазы 1 + нота; approve не гейтится.
6. **Nav-shell — НЕ строился** (см. «Отклонения»).

## Не сделано / отложено

- **Nav-shell (View-enum, таб-бар)** — вопрос ратификации №4 был «правильная ли это
  граница против оверинжиниринга»; при имплементации выяснилось: с одним видом
  таб-бар рендерил бы одну вкладку, а View-enum из одного варианта — мёртвая
  структура, которую Этап 3 всё равно перекроит под свои нужды (Receive знает, какие
  клавиши и какой layout ему нужны). По правилу №5 (не строить под ещё-не-написанный
  код) НЕ построил; Этап 3 введёт View сразу с двумя реальными вариантами. Это
  сужение ратифицированного скоупа — ЖДЁТ ПОДТВЕРЖДЕНИЯ Ревьюера/Капитана.

## Замечено, не трогаю — вне скоупа

- Свежий pty имеет размер 0×0 — фит-гейт честно глушит approve (обнаружено смоуком,
  поведение корректное, не баг).
- `main.rs` docstring v0.1→v0.2 обновлён по касанию (та же правка блока).

## Отклонения от плана и почему

1. **Nav-shell не построен** — см. выше. ЖДЁТ РАТИФИКАЦИИ.
2. **Клиентская блокировка approve/deny при lockout НЕ реализована** (спека
   обещала «submit не диспетчится»): серверный ladder блокирует ТОЛЬКО PIN-пути
   (auth, high-risk approve — протокол §4); обычный approve и deny залоченной
   сессии сервер разрешает. Клиентская блокировка запретила бы действия, которые
   сервер позволяет, — это была ошибка спеки. Реализовано честно: баннер
   информирует, submits живут, повторный `locked` ре-армит баннер. ЖДЁТ
   ПОДТВЕРЖДЕНИЯ.
3. **Отсчёт lockout-баннера статический** («Retry in ~Ns», значение сервера),
   не тикающий: тикающий требовал бы абсолютного времени в момент ответа, а у
   чистой модели нет часов (clock приходит только в render). Прокидывать now в
   `Msg::Reply` — рябь на 50+ тест-вызовов ради секундомера на информационном
   баннере. `~` в тексте честно маркирует приблизительность. ЖДЁТ ПОДТВЕРЖДЕНИЯ
   (спека обещала «обратный отсчёт»).
4. **`internal` от context не деградирует, а фаталит** (спека числила его в
   деградации): канон-протокол (после МИНОРа Ревьюера Этапа 1) фиксирует — context
   не отвечает `internal` вовсе; неизвестный код = аномалия канала → fail-closed
   Fatal, консистентно с parse_get/parse_approve.

## Как тестировал — команды + вывод

```
$ cargo test              → 137 + 15 + 1 passed, 0 failed
$ cargo clippy --all-targets -- -D warnings   → чисто
$ cargo fmt --check       → чисто
$ cargo deny check        → advisories/bans/licenses/sources ok
```

**Red-мутации (правило №0):**
- Мутация «терминальный ответ снова завершает сессию» → 8 тестов упали
  (7 резидентных app.rs + 1 ui.rs) → откат → зелёно.
- Мутация «From-блок не рендерится» → 1 тест упал (from_to full addresses) →
  откат → зелёно.

**Ре-балансировка замеренных 13 тестов**: 7 app.rs exit-семейства → 7 резидентных
(+2 новых lockout, +1 второе-решение-за-сессию); 4 main.rs → session-коды пиннятся
значениями, decision_line над Decision, drain-контракт; ui.rs resolved-экран →
outcome-баннер (+3 новых: lockout-баннер, From→To, деградация); protocol.rs hello
фикстура → proto:2 (+6 новых parse_context); transport.rs +2 (context round-trip,
wallet_locked не фатален).

**DoD-смоук** (реальный бинарь в pty против Python-стаба proto 2, 2 кейса ×
11-12 проверок — ВСЕ PASS): интерактив — From→To с полными адресами, APPROVED-баннер,
процесс жив после решения, поллинг продолжился, exit 6 на quit, decision-строки в
TUI нет; пайп — ровно одна JSON-строка `{"decision":"approved","tx_hash":"0xfeed"}`.
