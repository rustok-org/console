# Релизный поезд §8 — console v0.2.0 доезжает до пользователя

> Статус: **ПРЕДЛОЖЕНИЕ, ждёт Гейта-1.** Кода нет до двух подписей.
> Управляющие: ТЗ §8 (порядок поезда, ратифицирован), §9.5 (Фаза 2 закрыта 7/7 —
> предусловие выполнено), ADR `console/.claude/decisions/2026-07-12-invariant-7-decision-stream.md`
> (инвариант #7 эволюционировал: решения = JSON-строки на не-TTY stdout, exit-коды только
> сессионные 1/2/3/6), отметка §9.5 Этапа 2: «⚠️ mcp/tests/e2e всё ещё ждёт one-shot
> exit-коды против пина CONSOLE_IMAGE=v0.1.0 — обновить на релизном поезде §8».
> Команда: Капитан/Инженер/Ревьюер; связь только через Капитана.

## Цель — один абзац

Довезти Console v2 (Фаза 1 + Фаза 2, уже на main) до пользователя: выпустить
`rustok-console v0.2.0` (бамп + тег → GHCR-образ), перевести wallet-образ mcp на новый
пин и **переписать e2e-приёмку mcp под резидентную модель** (сейчас она ждёт one-shot
exit-коды и экраны v0.1 — с пином v0.2.0 упадёт вся). Пересборка `rustok-wallet-tui`
0.6→0.7 и листинги — ops Капитана (память: publish = ops Капитана), поезд их готовит.

## Замер blast radius (сделан ДО спеки; console `5ab40c8`, mcp `1f49180`)

1. **console**: `Cargo.toml` version 0.1.0→0.2.0 (+lock) — всё. `CLIENT_ID` собирается из
   `CARGO_PKG_VERSION` (informational, сервер не валидирует — канон §3.1). Тег `v0.2.0` →
   `docker-publish.yml` (триггер `tags: v*`) пушит `ghcr.io/rustok-org/rustok-console:{v0.2.0, v0.2, v0}`;
   `ci.yml` на теге гоняет обычные 4 джоба.
2. **mcp**: `Dockerfile.wallet` — 2 строки (ARG:19 + пример в комменте:10); `CHANGELOG.md`
   — запись; **тесты**: `tests/e2e/console.py` (драйвер, 149 строк — `wait_exit` жмёт
   Enter и ждёт СМЕРТИ процесса: в v2 это вечное ожидание) ·
   `tests/e2e/test_approval_e2e.py` (5 тестов: exit 0/4/1 = APPROVED/REJECTED/FAILED —
   в v2 не существуют) · `test_expiry_e2e.py` (финальный экран-якорь «Pending approvals: 0»
   — текст ушёл ещё на Этапе 3, home теперь Dashboard).
3. **mcp CI: e2e-джоба НЕТ** (проверено грепом workflows) — e2e гоняются локально на
   podman (маркеры `e2e`/`e2e_slow`). Следствие: mcp-PR не самоблокируется через CI;
   доказательство = локальный прогон e2e против собранного v0.2.0-образа (DoD).
4. Прочие v0.1.0-упоминания в mcp: только CHANGELOG-история (не трогаем) и
   Dockerfile.wallet (трогаем). `SKILL.md`/доки листингов — линия tui 0.6→0.7,
   ops Капитана, не этот PR.

## Стейл-карта e2e (что именно врёт про v2)

| Было (v0.1 one-shot) | Стало (v0.2 резидент, ADR #7) |
|---|---|
| решение → консоль выходит; exit 0=APPROVED, 4=REJECTED | решение → нотис APPROVED/REJECTED/FAILED, консоль ЖИВЁТ, возврат к очереди; выход только `q` → **6 (EXIT_ABORTED)** |
| failed broadcast → exit 1 (FATAL) | FAILED — нотис, НЕ фатал; exit по `q` → 6 |
| после PIN — очередь («Pending approvals») | после PIN — **Dashboard** («Waiting for you»); очередь через `a`, таб-бар «Queue·N [a]» |
| `wait_exit()` = Enter + ждать смерти процесса | Enter больше не «dismiss» (нотис не блокирует); нужен `quit()` = `q` + ждать exit |
| exit 3 NO_TTY | **без изменений** (сессионный код сохранён Гейтом-1 Этапа 2) |

## Скоуп — что входит / что ЯВНО не входит

**Входит (3 шага, строго по порядку):**

1. **console-PR «бамп»**: `Cargo.toml` 0.1.0→0.2.0 + `Cargo.lock`. После мержа — **тег
   `v0.2.0` на мерж-коммит** (пушу я по «добро» Капитана — публикация образа
   необратима) → дождаться зелёного `docker-publish` → проверить `ghcr` манифест тега.
2. **mcp-PR «пин + честная приёмка» (атомарный — пин и тесты вместе, иначе красное окно)**:
   - `Dockerfile.wallet`: ARG + коммент-пример → v0.2.0;
   - `tests/e2e/console.py`: `wait_exit` → честные `quit()` (q → exit) и `wait_exit_after(keys)`;
     навигационный хелпер «PIN → Dashboard → `a` → очередь» (v2-home);
   - `test_approval_e2e.py`: якоря/флоу v2 — approve: нотис APPROVED → **ассерт
     резидентности (процесс жив, очередь на экране)** → `wallet.status == executed` →
     `q` → exit 6; reject: REJECTED → denied → 6; double-spend: нотисы + статусы
     {executed, failed} через agent-RPC → 6 (НЕ exit 1); no-tty: 3 без изменений;
   - `test_expiry_e2e.py`: финальный якорь → v2: после PIN home=Dashboard, точный текст
     «Waiting for you: nothing pending» (ui.rs:595, проверено /check);
   - `CHANGELOG.md`: запись про пин v0.2.0 + резидентную модель.
   - **DoD-прогон**: conftest берёт образ из `RUSTOK_E2E_IMAGE` (или DEFAULT_IMAGE) —
     red/green гоняются чисто через env, без жонглирования Dockerfile: (red) новые
     v2-тесты против образа со СТАРЫМ пином — падают/висят по конструкции; (green)
     `podman build -f Dockerfile.wallet` с новым пином → `RUSTOK_E2E_IMAGE=… uv run
     pytest -m e2e` (7 сценариев, ~2 мин) зелёные; `e2e_slow` (15-мин expiry) — один
     раз (решение №4). Логи обоих прогонов в отчёт.
3. **Передача ops Капитану** (не мой код): пересборка `rustok-wallet-tui` 0.6→0.7
   (podman, GHCR-пуш) + листинги ClawHub/mcp-publisher/Smithery. Готовлю точный
   чеклист-комментарий в отчёте.

**ЯВНО НЕ входит:** новые e2e-сценарии под фичи Фазы 2 (Dashboard/Receive/Activity —
покрыты 242 тестами и смоуками console-репо; e2e mcp доказывает МОСТ агент↔человек, не
UI) · SKILL.md/доки листингов (линия tui, ops) · same-uid-целостность журнала (отложена
Капитаном) · core — ноль.

## Критерии приёмки — «поезд готов, когда…»

1. GHCR несёт `rustok-console:v0.2.0` (semver-теги v0.2.0/v0.2/v0 — метаданные
   docker-publish проверены /check; манифест и дата — после пуша тега).
1b. `grep -rn "0\.1\.0"` по console-докам/README — не осталось хардкодов версии
   (кроме исторических записей).
2. mcp-PR: `grep -rn "v0.1.0" Dockerfile.wallet tests/` → 0 вхождений; e2e-ассерты
   не содержат exit 0/4 и «Pending approvals».
3. Локальный e2e-прогон против нового образа: быстрые зелёные, `e2e_slow` зелёный
   (вывод pytest в отчёте).
4. Новые ассерты v2 включают **резидентность** (после APPROVED процесс жив) — это
   поведенческое сердце v2, не только смена кодов.
5. Гейты mcp-CI дословно (проверено /check: e2e в CI НЕТ — addopts исключает маркеры):
   `uv run ruff check src tests` · `uv run ruff format --check src tests` ·
   `uv run mypy src` · `uv run pytest` (юниты) · shellcheck — все зелёные локально.
6. Чеклист ops для Капитана (tui 0.7 + листинги) в отчёте, с точными командами.

## Решения к ратификации (Гейт-1) — предрешены

1. **Тег пушу я, по отдельному «добро» после мержа bump-PR** — (а) рекомендация: так;
   (б) тег = публикация образа, необратимый акт — двухшаговость дешевле отката GHCR;
   (в) контраргумент «тег мог бы пушить Капитан» — не перевесил: механика скриптовая,
   контроль остаётся через «добро»; (г) цена ошибки: удалить тег+образ (грязно, но

2. **mcp-PR атомарный (пин + e2e вместе)** — иначе между PR-ами живёт красная приёмка;
   e2e не в CI, но красная приёмка на main = ложь репозитория.
3. **Ассерты исходов — двухслойные**: экранный нотис (человеческая правда) +
   `wallet.status` через agent-RPC (машинная правда). Один слой без другого — полуправда

   дедлайна на РЕЛИЗНОМ образе (force_expire в релизе не существует — докстринг теста);

## Тест-план

Сам PR — это тесты. Red→green через `RUSTOK_E2E_IMAGE`: (red) новые v2-тесты против
old-pin-образа падают; (green) против v0.2.0-образа зелёные. Негативный контроль:
один старый ассерт (`wait_exit` после решения) против v0.2.0 таймаутится с экраном
живой консоли — доказательство, что переписывание было обязательным (вывод в отчёт,
тест в PR не остаётся). Экранные якоря v2: «Waiting for you: nothing pending» ·
таб «Queue·N [a]» · нотисы APPROVED/REJECTED/FAILED (слова notice_line, ui.rs:96-99).
