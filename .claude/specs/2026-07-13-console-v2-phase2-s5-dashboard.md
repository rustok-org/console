> Статус: **ПРЕДЛОЖЕНИЕ, ждёт Гейта-1.** Кода нет до «го».
> Управляющие: ТЗ §2/§5/§9.5 (Дашборд после core PR2 ✅), план Фазы 2 («PIN-unlock →
> Dashboard»), канон `docs/APPROVER-PROTOCOL.md` §3.8 (`positions` + **нормативное
> клиентское правило**: не слать positions при открытой карточке; приоритет `list`).
> **Обязательство спеки (флаг Ревьюера, Гейт-1 core PR2, зафиксирован в ТЗ §9.5):**
> клиентский шедулер даёт `list` приоритет над `positions`/`activity` — медленный

> с тикающим дедлайном.

## Цель — один абзац

Третий вид консоли — **Dashboard**: баланс по чейнам (из `context`), DeFi-позиции (новый
клиентский `positions`-op), «Waiting for you: N». Ядро этапа — не вёрстка, а **read-op
шедулер** Model: политика, при которой приватные чтения никогда не конкурируют с очередью
за единственный слот. Активность на дашборде НЕ рендерится до Этапа 7 (op не существует;
заглушек не строим — ратифицировано).

## Скоуп PR — что входит / что ЯВНО не входит

**Входит (репо console целиком; core/канон НЕ трогаются):**

1. **protocol.rs**: `Request::Positions`; клиентский тип `Position` — зеркало канона §3.8
   (9 полей; `extra: BTreeMap<String,String>` — **рендерим, не парсим**: `"∞"`, `"80%"` —
   дисплей-строки по канону); `PositionsOutcome { Ok(Vec<Position>) | WalletLocked }`;
   parse: `ok` → массив (пустой валиден — best-effort канона), `wallet_locked` →
   деградация, `unauthorized`/`protocol_error` → ProtocolError (fail-closed класс, как
   у context); мусор → Malformed.
2. **transport.rs**: прокладка Request/Reply (механически, паттерн Context).
3. **app.rs — read-op шедулер (ядро):**
   - Model — новые поля, механизм каждого назван явно (Гейт-1): `positions:
     Positions` (тристейт /check-4: `NotYet | Loaded(Vec<Position>) | Unavailable`),
     `read_age: u32` (генерическая свежесть read-данных в ТИКАХ — у чистой модели нет
     часов; урок Этапа 2), `next_read: ReadOp` (флаг чередования Context/Positions),
     **`context_stale: bool`** — ставится, когда РЕФРЕШ context ответил
     `wallet_locked` (старые данные живы, но помечены), сбрасывается успешным
     рефрешем; именно он питает ноту Balance-блока — не выводится из `read_age`.
   - **Политика list-first (канон §3.8 + флаг Ревьюера), нормативно:**
     read-op (`Context`-рефреш | `Positions`) уходит в сокет ТОЛЬКО из `on_reply(List)` —
     сразу после свежего ответа очереди, когда слот гарантированно только что обслужил
     `list`; НИКОГДА из `on_tick` (тик — собственность `list`). Условия: view ==
     Dashboard, `confirm.is_none() && !awaiting_card && pending.is_none()`, данные
     стейл (`read_age >= STALE_TICKS`). **Порядок диспетча в on_reply нормативен
     (/check-1): `flush_pending()` — пользовательский интент — ПЕРВЫЙ; read-op
     рассматривается только когда flush ничего не вернул.** Red-тест: park get во
     время list-RTT при стейл-дашборде → после List уходит Get, не Positions.
     Инвариант, проверяемый тестом, — **порядок происхождения** (формулировка
     Ревьюера, Гейт-1): read-op никогда не встаёт ПЕРЕД назревшим `list` — после
     любого тика при свободном слоте первым уходит `List`, никогда read-op.
     **Честная граница гарантии:** если read-op уже ушёл и не отвечает (канон §3.8 —
     серверного таймаута нет by design), list-поллинг стоит вместе с ним — это ТОТ ЖЕ
     класс риска, что принят для `context` с Этапа 2 (первый фетч после auth), не
     новый; Этап 5 повышает частоту окна (~каждые 30 c активного дашборда), поэтому
     риск назван здесь явно, а не унаследован молча. Полное закрытие (дедлайн на
     серверном `track()` или клиентский reconnect-по-таймауту) — core-правка или
     протокол-изменение, за пределами этого PR; идея «dashboard data may be delayed»
     индикатора — кандидат в бэклог, не строим сейчас. Повторные входы на Dashboard
     помечают стейл — рефреш на вход bounded структурно: максимум один read-op на один
     list-reply (/check-5, осознанно).
   - **Рефреш обоих**: `Positions` и `Context` (балансы) чередованием на том же шедулере
     (решение №2 ниже) — дашборд с балансом на момент входа «врёт» через час жизни
     резидентной консоли.
   - `STALE_TICKS = 12` (~30 c при POLL 2.5 c — позиции = живые eth_call, не дёргаем
     каждый тик); вход на Dashboard помечает данные стейл (первый показ — сразу после
     ближайшего list-reply).
   - Деградации: positions — **тристейт** (/check-4): `NotYet` («loading
     positions…», первый вход до первого ответа) | `Loaded(Vec)` | `Unavailable`
     («positions unavailable», `wallet_locked`); ошибка парса — ProtocolError → Fatal
     (fail-closed, класс context). **Рефреш context, ответивший `wallet_locked`
     (/check-3): старые данные НЕ выбрасываются** (From→To карточки продолжает жить),
     но Balance-блок дашборда получает ноту «balance may be stale — refresh failed»;
     флаг staleness сбрасывается успешным рефрешем; тест.
4. **ui.rs**: вкладка ` Dashboard [d] ` (третья); `render_dashboard`: блок Balance
   (per-chain: `chain_id` + `format::wei_to_eth(balance)` — bignum-safe, человечески),
   блок Positions (строка на позицию: `protocol · asset_symbol · balance_formatted
   asset_name` + extra-пары verbatim ключ-значение; **адресов активов на дашборде нет
   вовсе** — решение №4), блок «Waiting for you: N — press a». Честность: позиции не
   влезли по высоте → маркер «+N more — terminal too small» (паттерн raw_data), никакого
   молчаливого клипа; деградации: без context — «balance unavailable», positions None —
   «positions unavailable», пустые — «no DeFi positions».
5. **main.rs**: map_key — Queue: +`d`; Receive: +`d`; Dashboard-арм: `a`/Esc → Queue,
   `r` → Receive, `q` quit, остальное мертво. `Msg::View(Dashboard)` — тот же тройной
   гейт (confirm/awaiting/parked-Get; red-тест).
6. **Домашний вид после auth — Dashboard** (решение №1, буква плана Фазы 2).
7. **Тесты (red→green):** парс positions (ok/пусто/wallet_locked/unauthorized/мусор +
   extra verbatim «∞»/«80%»); шедулер — красные мутации: (а) read-op из on_tick →
   инвариант-тест падает; (б) read-op при открытой карточке/awaiting/parked → падает;
   (в) каденция STALE_TICKS; (г) чередование Context/Positions; (д) **порядок
   происхождения** (read-op не встаёт перед назревшим list — переименован честно,
   Гейт-1); ui-снапшоты Dashboard (полный/деградации×3/overflow-маркер/третья вкладка
   активна); map_key-армы; тройной гейт для Dashboard.

**ЯВНО НЕ входит:** секция активности на дашборде (Этап 7; НЕ рендерим вовсе, не
заглушка); `activity`-op; возврат `short_addr` (бэклог; адреса на дашборде не
показываются); core/канон-правки; realtime push; сортировки/фильтры позиций
(серверный порядок детерминирован); USD-оценка (оракула нет by design).

## Решения к ратификации (Гейт-1) — предрешены

1. **Home после auth = Dashboard** — (а) буква плана Фазы 2 («PIN-unlock → Dashboard»),
   продукт «пульт кошелька»; (б) «Waiting for you: N» + счётчик в табе ведут к очереди
   одним `a`; (в) контраргумент: лишний кейпресс до approve при горящем дедлайне — не
   перевесил: list-поллинг жив из любого вида, дедлайны видны счётчиком, а карточка и
   раньше требовала enter из очереди; (г) цена ошибки — один дефолт enum, обратимо;
   (д) переключатель: слово Капитана (ТВОЙ ВЫБОР — продуктовое).
2. **Рефреш context (балансов) чередованием с positions** — (а) резидентный дашборд со
   снапшотом баланса от auth «врёт»; (б) тот же list-first слот, каденция та же;
   (в) контраргумент: больше RPC на ядре — не перевесил: раз в ~60 c на op при активном
   дашборде; (г) цена: обратимо флажком в шедулере; (д) переключатель: Ревьюер сочтёт
   session-snapshot достаточным.
3. **STALE_TICKS = 12 (~30 c)** — живые eth_call не дёргаем каждый тик; переключатель —
   любое слово Капитана по вкусу продукта.
4. **Адресов активов на дашборде нет** (symbol/name достаточно) — сокращать нельзя
   (ТЗ §4.1 разрешает short только на дисплей-списках, но `short_addr` удалена в Фазе 1
   как dead-code и её возврат — отдельный бэклог); полные 42-символьные адреса на
   каждой строке позиций — шум. Переключатель: продуктовое слово.
